# 선거 후보자 프로필 및 공약 홍보 웹페이지 PRD

## 1. 문서 개요

- 프로젝트명: 선거 후보자 프로필 및 공약 홍보 웹페이지 구축
- 작성일: 2026-03-15
- 대상 폴더: `wishket/선거후보자프로필및공약홍보웹페이지구축`
- 문서 목적: `OVERVIEW.md` 기준으로 실제 구축 범위, 정보 구조, 필수 기능, 운영 전제를 명확히 정의한다.
- 권장 구축안: **모바일 우선 반응형 단일 스크롤 페이지 1개**를 MVP로 제안하고, 공약 분량이 많을 경우 `공약 상세` 보조 페이지 1개를 선택 확장한다.

## 2. 프로젝트 배경

- 본 프로젝트는 선거 기간 중 후보자의 프로필, 핵심 메시지, 대표 공약, 활동 이미지를 빠르게 전달하는 공식 홍보용 웹페이지다.
- 예산과 일정이 매우 짧기 때문에 기능을 넓게 확장하는 방식보다, **읽기 쉬운 정보 구조와 빠른 제작/수정 가능성**이 우선이다.
- 사용자는 대부분 모바일에서 링크, 문자, 카카오톡, SNS, 지역 커뮤니티를 통해 유입될 가능성이 높다.
- 따라서 첫 화면에서 후보자 인지, 다음 스크롤에서 공약 이해, 마지막 구간에서 공식 채널 이동까지 자연스럽게 이어지는 구조가 필요하다.

## 3. 제품 목표

### 3.1 비즈니스 목표

- 후보자의 신뢰감과 인지도를 짧은 시간 안에 전달한다.
- 유권자가 대표 공약을 쉽게 이해하도록 만든다.
- 선거 기간 동안 공식 정보 허브 역할을 수행한다.
- 향후 인쇄물, SNS, 문자 링크에서 반복 활용 가능한 대표 랜딩 페이지를 확보한다.

### 3.2 제품 목표

- 첫 화면 15초 이내에 후보자 이름, 직함/출마 정보, 핵심 슬로건, 대표 CTA가 인지되어야 한다.
- 첫 2스크롤 안에 프로필과 핵심 공약 3~5개가 모두 보여야 한다.
- 360px 모바일부터 1440px 이상 데스크톱까지 정보 위계가 무너지지 않아야 한다.
- 텍스트/사진 교체가 쉬운 구조여야 하며, 운영자가 문구 수정 요청 시 빠르게 반영할 수 있어야 한다.

## 4. 비목표

- CMS/관리자 페이지 구축
- 게시판, 댓글, 회원가입, 로그인
- 실시간 후원 결제 기능
- 뉴스 아카이브 대규모 관리 기능
- 다국어
- 대규모 애니메이션 연출, 영상 중심 사이트

## 5. 핵심 사용자

### 5.1 1차 사용자

- 후보자 정보를 빠르게 확인하려는 일반 유권자
- 공약만 빠르게 비교하고 싶은 모바일 사용자
- 사진과 활동 이력을 보고 신뢰성을 확인하려는 지역 주민

### 5.2 2차 사용자

- 후보 캠프 실무자
- 보좌 인력 또는 홍보 담당자
- 링크를 공유받아 공식 정보 여부를 검증하려는 관계자

## 6. 핵심 시나리오

### 시나리오 A. 링크 유입 후 빠른 인지

1. 사용자가 문자, SNS, 카카오톡 링크로 사이트에 진입한다.
2. 첫 화면에서 후보자 사진, 이름, 출마 정보, 핵심 문장을 본다.
3. 바로 아래에서 프로필 요약과 대표 공약을 확인한다.
4. 필요 시 공식 연락 채널 또는 SNS로 이동한다.

### 시나리오 B. 공약 집중 탐색

1. 사용자가 특정 이슈 때문에 사이트에 진입한다.
2. 공약 카테고리 또는 핵심 공약 목록을 먼저 확인한다.
3. 상세 설명 아코디언 또는 보조 페이지에서 내용을 읽는다.
4. 활동 사진/현장 메시지를 통해 후보자 톤을 재확인한다.

### 시나리오 C. 캠프 운영 업데이트

1. 캠프 담당자가 사진, 문구, 공약 순서를 수정 요청한다.
2. 개발자는 정적 데이터 파일 기준으로 빠르게 교체한다.
3. 배포 후 공유 썸네일과 모바일 레이아웃까지 함께 검수한다.

## 7. 권장 정보 구조

### 7.1 MVP 권장 구조

```text
/
  히어로
  후보자 핵심 소개
  프로필 / 이력
  핵심 공약 요약
  공약 상세
  활동 사진 / 현장 스케치
  공식 채널 / 연락처 / 공유 CTA
  푸터
```

### 7.2 선택 확장 구조

```text
/
  메인 스크롤 페이지
/pledges
  공약 상세 모음
```

### 7.3 IA 방향

- 기본은 **1페이지 완결형 구조**로 간다.
- 공약이 8개 이상이거나 상세 텍스트가 길 경우에만 `/pledges` 보조 페이지를 연다.
- 갤러리만 별도 페이지로 분리하는 방식은 현재 일정과 예산에서는 우선순위가 낮다.

| 후보자 소개 | 인간적인 인상과 한 줄 설명 전달 | 한 줄 소개, 핵심 가치 3개, 대표 메시지 |

| 연락/채널 | 공식 채널 이동 유도 | 전화, 카카오채널, SNS, 선거사무소 주소, 지도 여부 |

## 9. 기능 요구사항

### 9.1 공통

- 모바일 우선 반응형 UI
- 상단 고정 헤더 또는 섹션 점프 내비게이션
- 섹션 앵커 이동
- 후보자 대표 이미지 최적화
- 기본 SEO 메타데이터와 OG 이미지 적용
- 카카오톡/문자/SNS 공유 시 깨지지 않는 썸네일 구성

### 9.2 히어로

- 후보자명, 출마 정보, 대표 슬로건 노출
- 대표 CTA 2개 배치
- 권장 CTA:
  - `핵심 공약 보기`
  - `후보자 소개 보기`
- 필요 시 보조 CTA로 공식 연락 채널 또는 SNS 링크 배치

### 9.3 후보자 프로필

- 사진 옆 또는 아래에 핵심 이력 카드 배치
- 긴 이력은 모두 노출하지 않고 대표 경력 중심으로 압축
- 모바일에서는 연표보다 카드형 요약 우선

### 9.4 공약 섹션

- 대표 공약 3~5개를 우선 노출
- 카테고리 탭 또는 필터형 탐색 가능
- 상세 설명은 아코디언 또는 확장 카드로 제공
- 각 공약마다 `무엇을`, `왜`, `누구를 위해`, `기대 효과` 구조 유지

### 9.5 활동 사진 / 갤러리

- 단순 썸네일 나열보다 캡션을 함께 제공
- 모바일에서는 1열 또는 가로 스크롤, 데스크톱에서는 2~3열 그리드
- 사진 수가 적을 경우 큰 카드 중심 편집형 레이아웃 사용

### 9.6 공식 채널 / 연락 영역

- 캠프에서 실제 사용하는 채널만 노출
- 전화, 문자, 카카오채널, SNS, 지도 링크 중 확정된 항목만 적용
- 법적 고지나 표기 방식은 캠프 최종 검수 기준으로 반영

## 10. 콘텐츠 운영 요구사항

### 10.1 필수 수집 자료

- 후보자 공식 사진 원본
- 후보자 이름/직함/출마 지역 표기 확정안
- 핵심 공약 3~5개와 상세 설명
- 프로필 상세 정보
- 활동 사진 6장 이상 권장
- 공식 연락처, 주소, SNS 링크
- OG 공유용 대표 문구

### 10.2 편집 원칙

- 문장 길이는 짧게 유지하고, 본문은 한 화면에서 훑어볼 수 있게 쪼갠다.
- 공약은 선언형 문장보다 구체 명사 중심으로 요약한다.
- 사진은 화질보다 표정과 상황이 분명한 컷을 우선한다.
- 선거 관련 필수 고지 문구는 캠프와 별도 검수 후 반영한다.

## 11. 반응형 및 접근성 요구사항

### 11.1 반응형 기준

- 모바일: 360px~767px
- 태블릿: 768px~1023px
- 데스크톱: 1024px~1440px+

### 11.2 반응형 원칙

- 첫 화면 핵심 정보는 모바일에서도 한 화면 안에 최대한 들어오게 구성한다.
- 프로필, 공약, 갤러리는 모바일 1열 기준으로 설계한다.
- 데스크톱에서는 타이포그래피와 사진의 비대칭 편집형 레이아웃을 사용한다.

### 11.3 접근성

- 모든 이미지에 의미 있는 대체 텍스트 제공
- 충분한 명도 대비 확보
- 키보드 포커스 표시
- 헤딩 구조를 논리적으로 유지
- 모바일에서 44px 이상의 터치 타깃 확보

## 12. SEO / 공유 / 계측

### 12.1 SEO

- `title`, `description`, OG, Twitter Card 적용
- 후보자명 + 선거명 + 지역명 조합을 기본 메타 규칙으로 사용
- 이미지 ALT와 섹션 제목을 검색 친화적으로 구성

### 12.2 공유

- 카카오톡, 문자, SNS 미리보기용 대표 썸네일 1종 제작
- 대표 슬로건과 후보자명은 썸네일 안에서도 분명히 읽혀야 한다.

### 12.3 기본 이벤트 계측

- `cta_click`
- `section_view`
- `contact_click`
- `share_click`

## 13. 기술 방향

- 프레임워크: Next.js App Router
- 언어: TypeScript
- 스타일링: Tailwind CSS + 프로젝트 토큰
- 폰트: Pretendard Variable
- 배포: Vercel 또는 정적 호스팅
- 데이터 입력 방식: 파일 기반 정적 데이터 또는 간단한 JSON/TS 객체

### 13.1 구현 원칙

- 짧은 일정에 맞춰 **백엔드 없이 빠르게 오픈 가능한 구조**를 채택한다.
- 텍스트와 이미지 수정이 쉬운 컴포넌트 구조를 우선한다.
- 애니메이션보다 가독성과 안정성을 우선한다.

## 15. 수용 기준

- 모바일, 태블릿, 데스크톱에서 주요 섹션이 모두 정상 동작해야 한다.
- 후보자 소개, 프로필, 핵심 공약, 활동 사진, 공식 채널이 모두 포함되어야 한다.
- Pretendard 기반 타이포그래피가 전 구간에 일관 적용되어야 한다.
- 프로젝트 전용 디자인 시스템과 중립적 적청 포인트가 적용되어야 한다.
- 공약 섹션은 3개 이상 등록 가능하고, 구조가 반복 가능해야 한다.
- 문구/사진 교체가 쉬운 구조여야 한다.
