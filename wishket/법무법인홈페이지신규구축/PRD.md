# 법무법인 홈페이지 신규 구축 PRD

## 1. 문서 개요

- 프로젝트명: 법무법인 형사 특화 홈페이지 신규 구축
- 대상 폴더: `wishket/법무법인홈페이지신규구축`
- 구현 앱: `wishket/법무법인홈페이지신규구축/proposal-demo`
- 문서 목적: 제안 단계 데모를 실제 구축형 프로토타입 수준으로 정리하고, 개발 범위와 정보 구조를 명확히 정의한다.

## 2. 프로젝트 배경

기존 홈페이지 콘텐츠를 활용하되, 성범죄와 마약 분야에 특화된 신뢰형 홈페이지를 새로 구축한다. 단순 회사 소개가 아니라 "전문성 전달 -> 긴급성 인식 -> 상담 전환" 흐름이 자연스럽게 이어져야 하며, 운영자는 성공사례와 전문분야 콘텐츠, 상담 접수를 직접 관리할 수 있어야 한다.

## 3. 목표

### 비즈니스 목표

- 특정 형사 사건 의뢰 잠재 고객에게 즉시 신뢰를 형성한다.
- 전문분야 페이지를 통해 검색 유입과 상담 전환을 동시에 만든다.
- 성공사례를 체계적으로 노출해 전문성과 대응 경험을 증명한다.
- 운영자가 CMS에서 콘텐츠와 상담 상태를 쉽게 관리할 수 있게 한다.

### 제품 목표

- 반응형 기준에서 정보 우선순위가 무너지지 않는 구조를 만든다.
- 메인, 전문분야, 성공사례, 상담, 운영 화면이 하나의 일관된 UX로 연결되게 한다.
- 제안용 단일 페이지가 아닌, 실제 서비스 IA를 반영한 다중 라우트 구조로 만든다.

## 4. 핵심 KPI

- 상담 CTA 클릭률
- 온라인 상담 접수 완료율
- 전문분야 상세 페이지 체류 시간
- 성공사례 상세 열람 수
- 신규 상담의 당일 1차 응답률
- CMS 내 성공사례 게시 업데이트 소요 시간

## 5. 주요 사용자

### 1차 사용자

- 성범죄 사건 대응이 필요한 잠재 의뢰인
- 마약 사건 대응이 필요한 잠재 의뢰인
- 보호자 또는 가족 대리 검색 사용자

### 2차 사용자

- 법무법인 운영 담당자
- 콘텐츠 관리자
- 상담 접수 담당자

## 6. 사용자 시나리오

### 시나리오 A: 긴급 상담 유입

1. 사용자가 검색 또는 광고를 통해 전문분야 상세 페이지에 들어온다.
2. 자신의 상황과 유사한 사건 유형, 대응 절차, FAQ를 빠르게 확인한다.
3. 즉시 전화 또는 온라인 상담 폼으로 전환한다.
4. 운영자는 접수 대기열에서 긴급도를 보고 우선 처리한다.

### 시나리오 B: 신뢰 검증형 탐색

1. 사용자가 메인 페이지에서 법무법인 소개와 대표 변호사 이력을 확인한다.
2. 성공사례 아카이브에서 결과 태그와 분야를 기준으로 사례를 탐색한다.
3. 상세 사례에서 대응 전략과 결과를 읽고 상담 신청으로 이동한다.

### 시나리오 C: 운영자 게시 관리

1. 운영자가 관리자 화면에서 성공사례, 전문분야, FAQ, 상담 대기 현황을 확인한다.
2. 비공개 검수 상태를 점검하고 공개 여부를 변경한다.
3. 상담 신청을 담당 변호사에게 배정하고 메모를 남긴다.

## 9. 기능 요구사항

### 9.1 메인

- 전문분야 바로가기 카드 제공
- 대표 상담 CTA 상단/중단/하단 반복 노출
- 변호사 신뢰 요소와 핵심 대응 원칙 요약
- 성공사례 일부 미리보기
- 상담 절차 안내

### 9.2 법무법인 소개

- 법무법인 소개 문구
- 운영 철학 및 대응 원칙
- 대표 및 담당 변호사 프로필
- 연혁/주요 이력

### 9.4 성공사례 아카이브

- 검색 기능
- 분야 필터
- 결과 태그 필터
- 카드형 목록
- 사례 상세 진입
- 익명화/비공개 관련 운영 정보 표현

### 9.5 성공사례 상세

- 사건 개요
- 대응 포인트
- 결과 및 의미
- 담당 변호사 정보

- 상담 CTA

### 9.6 온라인 상담

- 사건 분야 선택
- 긴급도 선택
- 선호 상담 방식 선택
- 사건 개요 입력
- 개인정보 동의
- 접수 후 triage 결과 미리보기
- API 기반 접수 처리와 운영 대시보드 반영

### 9.7 운영/CMS 대시보드

- 오늘 처리해야 할 상담 건수
- 긴급 상담 큐
- 성공사례 공개 상태
- 전문분야 콘텐츠 검수 상태
- 게시 캘린더와 릴리즈 체크리스트

## 11. 비기능 요구사항

- 반응형 기준: 모바일 360px 이상, 데스크톱 1280px 기준에서 구조 유지
- 성능: 정보 페이지 첫 화면에서 과도한 애니메이션/대용량 자산 지양
- 접근성: 명확한 heading 구조, 충분한 대비, 폼 label 제공
- SEO: 전문분야/성공사례 상세에 개별 메타데이터 적용 가능 구조
- 유지보수성: 목업 데이터와 페이지 컴포넌트를 분리해 이후 CMS 연동이 쉬워야 함

## 12. 기술 제안

- 프레임워크: Next.js App Router
- 언어: TypeScript
- 스타일링: Tailwind CSS + 글로벌 토큰
- 아이콘: lucide-react
- 향후 확장 가정:
  - DB/CMS: Supabase 또는 Headless CMS
  - 배포: Vercel
  - 폼 저장: Supabase/Postgres 또는 별도 백엔드 API

## 13. 수용 기준

- 각 주요 메뉴가 실제 라우트로 분리되어야 한다.
- 전문분야와 성공사례는 상세 페이지까지 이동 가능해야 한다.
- 성공사례 목록은 검색/필터가 동작해야 한다.
- 상담 폼은 입력과 제출 후 상태 변화가 보여야 한다.
- 운영 화면에서 상담 큐와 콘텐츠 상태를 실제 운영 도구처럼 읽을 수 있어야 한다.
- `npm run lint`, `npm run build`를 통과해야 한다.
