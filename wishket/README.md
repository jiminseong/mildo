# Wishket Workspace

위시켓 지원서 초안, 고객별 메모, 제출용 문서를 모아두는 폴더입니다.

## Rules

- 앱 소스에서 `wishket/` 아래 파일을 import 하지 않습니다.
- 배포 업로드 대상에서 제외되도록 루트 `.vercelignore`에 `wishket/`를 등록합니다.
- 고객별 문서는 프로젝트명 또는 지원 공고명 기준으로 폴더를 나눠 저장합니다.
- 지원서 본문에는 이메일, 전화번호, 회사명 등 직접 거래 유도 정보는 넣지 않습니다.

## Recommended Structure

```text
wishket/
  README.md
  client-or-project-name/
    application.md
    notes.md
    links.md
```

## Wishket Form

위시켓 지원서 작성 시 기본 입력 항목은 아래 순서를 기준으로 정리합니다.

### 1. 지원 금액

- 분석한 업무 범위를 기준으로 실제 진행 가능한 금액을 제안합니다.
- 위시켓 수수료와 세금을 감안해 실제 수령 금액도 따로 계산합니다.
- 고정가인지, 범위에 따라 조정 가능한지 내부 메모에 남깁니다.

### 2. 지원 기간

- 실제 진행 가능한 기간을 `일` 단위로 제안합니다.
- 최대 4자리 기준으로 작성합니다.
- 클라이언트 예상 기간보다 짧게 제안하면 가능한 이유를 본문에 같이 적습니다.

### 3. 지원 내용

아래 3개 블록을 기본 템플릿으로 사용합니다.

```text
<프로젝트 진행 제안>
- 요구사항 분석 결과
- 작업 범위 정의
- 디자인/개발 순서
- 주차 또는 단계별 일정
- 검수 및 수정 반영 방식

<유사 프로젝트 진행 경험>
- 관련 업종 또는 기능 경험
- 실제 구현했던 핵심 포인트
- 이번 프로젝트에 바로 적용 가능한 경험

<사용 기술과 툴>
- 사용할 기술 스택
- 작업 방식
- 커뮤니케이션 방식
- 산출물 정리 방식
```

지원 내용 작성 체크:

- 첫 문단에서 프로젝트 이해도를 바로 보여줍니다.
- 구현 범위와 제외 범위를 구분해서 씁니다.
- 클라이언트가 궁금해할 리스크, 일정, 커뮤니케이션 방식을 같이 적습니다.
- 과장보다 실제 납품 가능한 범위를 우선합니다.

## Support Writing Tips

위시켓 아티클 `[파트너스] 지원 내용 작성법 - 좋은 지원 내용 vs 나쁜 지원 내용 비교`(2019-03-14) 요약:

### 1. 프로젝트 관심을 먼저 보여줍니다

- `내용 확인했습니다. 지원합니다.` 같은 문장만 쓰면 프로젝트를 읽지 않은 지원서처럼 보입니다.
- 첫 문단에서 프로젝트를 어떻게 이해했는지, 어떤 기능과 흐름이 핵심인지 바로 적습니다.
- 예상 이슈, 확인이 필요한 논점, 제안하고 싶은 구현 방향이 있으면 짧게라도 포함합니다.
- 사용하는 기술과 툴은 이름만 적지 말고 왜 적합한지도 같이 설명합니다.

좋은 방향 예시:

```text
- 현재 공고 기준으로 핵심은 회원 흐름, 관리자 승인, 반응형 웹 운영으로 이해했습니다.
- 이 프로젝트는 메인 기능 수보다 상태값과 권한 설계가 더 중요한 유형으로 보입니다.
- 미팅 전 기획 문서 확인이 가능하다면 일정과 범위를 더 정확하게 제안드릴 수 있습니다.
```

### 2. 관련 경험은 적게, 구체적으로 씁니다

- `다양한 프로젝트 경험`, `최고급 기술`, `만족스러운 결과` 같은 표현은 신뢰를 주기 어렵습니다.
- 관련 경험은 1~2개만 고르고, 어떤 기능을 구현했는지와 어떤 문제를 해결했는지 구체적으로 씁니다.
- `무엇을 맡았는지`, `어떤 기술을 썼는지`, `이번 프로젝트에 어떻게 바로 적용되는지`까지 연결합니다.
- 직접적인 유사 업종 경험이 없으면, 구현 방식 제안과 구조 이해도를 근거로 전문성을 보여줍니다.

나쁜 표현:

```text
안녕하세요. 다양한 분야의 플랫폼과 애플리케이션을 다수 진행했습니다.
즉시 착수 가능하며 만족스러운 결과를 보장해드리겠습니다.
```

좋은 표현 방향:

```text
- 사용자 인증, 관리자 권한 분리, 단계형 데이터 저장이 필요한 서비스는 직접 설계·구현한 경험이 있습니다.
- 반응형 웹 구축 프로젝트에서는 기획, 디자인, 개발을 함께 진행하며 실제 운영 가능한 구조로 납품했습니다.
- 이번 프로젝트에서도 학생용 흐름과 관리자 승인 흐름을 분리해 복잡도를 관리하는 방식으로 접근할 수 있습니다.
```

### 3. 업무 프로세스와 견적 근거를 제안합니다

- `자세한 건 미팅에서`, `견적은 지금 알기 어렵습니다`만 쓰면 수행 방식이 보이지 않습니다.
- 현재 공개된 범위를 기준으로라도 작업 단계, 예상 일정, 검수 방식을 먼저 제안합니다.
- 금액과 일정이 조정 가능하다면 왜 그런지 같이 설명합니다.
- 문서가 부족한 공고라도 `현 범위 기준 추정치`라는 전제를 달면 충분히 설득력 있게 쓸 수 있습니다.

좋은 방향 예시:

```text
- 1주차: 요구사항 및 정책 정리
- 2주차: 화면 설계 및 데이터 구조 확정
- 3~6주차: 핵심 기능 구현
- 7~8주차: QA, 수정, 배포
- 현재 견적은 공고 범위 기준이며, 미팅 후 상세 범위에 따라 조정 가능합니다.
```

### 4. 복붙 티가 나는 문장은 피합니다

- 프로젝트명, 핵심 기능, 타깃 사용자, 예상 리스크가 하나도 없는 지원서는 복붙으로 읽히기 쉽습니다.
- 지원할 때마다 전부 새로 쓰지 않더라도, 첫 문단과 관련 경험 문단은 공고에 맞게 반드시 바꿉니다.
- `왜 이 프로젝트에 지원하는지`와 `왜 내가 적합한지`가 한 번은 분명히 드러나야 합니다.

### 5. 제출 전 최종 체크

- 첫 3줄 안에 프로젝트 이해도가 드러나는가
- 관련 경험이 추상적 표현이 아니라 기능/역할/해결 방식으로 적혀 있는가
- 금액과 기간에 대한 기준이나 전제가 적혀 있는가
- 구현 범위와 별도 협의 범위가 구분되어 있는가
- 미팅 전에 확인하고 싶은 논점이나 질문이 있는가
- 어느 프로젝트에도 붙일 수 있는 문장처럼 보이지 않는가

## Pricing Strategy

위시켓 아티클 `[파트너스] 프로젝트 지원할 때 견적이 낮으면 유리할까요?`(2020-03-10) 요약:

- 낮은 금액 자체가 선정의 최우선 기준은 아닙니다.
- 위시켓이 소개한 2019년 기준 데이터에서는, 클라이언트 예상 금액보다 낮게 지원해 미팅 후 최종 계약도 더 낮은 금액으로 체결된 경우가 14% 수준이었다고 설명합니다.
- 클라이언트는 너무 낮은 금액을 보면 `품질이 불안하다`, `공수 산정이 잘못된 것 같다`고 느낄 수 있습니다.
- 금액보다 더 크게 보는 요소는 유사 프로젝트 경험, 빠른 피드백, 지속적인 관계 유지 가능성입니다.

실무 적용 기준:

- 무조건 최저가로 들어가지 않습니다.
- 클라이언트 예산 안에서 `실제로 납품 가능한 금액`을 제안합니다.
- 낮게 제안해야 할 이유가 분명할 때만 낮춥니다.
- 금액을 낮출수록 지원 내용과 포트폴리오에서 신뢰를 더 강하게 만들어야 합니다.

견적 산정 시 내부 체크:

- 위시켓 수수료를 제외하고도 남는가
- 세금과 기타 비용을 제외해도 작업 시간이 성립하는가
- 기획/디자인/개발/QA/커뮤니케이션 시간을 모두 포함했는가
- 수정 대응 버퍼와 운영 문의 대응 시간을 반영했는가
- 미팅 후 범위 확장 가능성을 감당할 여지가 있는가

권장 방식:

- 외부 제출 금액은 `실행 가능한 금액`으로 씁니다.
- 내부 메모에는 `최소 수락 가능 금액`, `희망 금액`, `수수료 제외 예상 실수령액`을 같이 적습니다.
- 클라이언트 예산보다 낮게 제안한다면 이유를 한 줄이라도 붙입니다.
- 예산이 너무 낮다고 판단되면 억지로 맞추지 말고, MVP 범위를 줄여서 같은 금액 안에 맞추는 방향을 먼저 제안합니다.

예시:

```text
- 현재 예산 기준에서는 1차 구축 범위를 MVP로 좁혀야 안정적인 납품이 가능합니다.
- 지원 금액은 현 범위 기준이며, 상세 요구사항 확인 후 조정될 수 있습니다.
- 예산을 더 낮추기보다는 관리자 기능/통계/고급 운영 기능을 2차 범위로 분리하는 것이 적절합니다.
```

하지 말 것:

- 이유 없이 예산보다 과하게 낮게 지원하기
- 수수료와 세금을 고려하지 않고 표면 금액만 보고 지원하기
- 실제로는 불가능한 기간과 금액을 미팅 유도용으로 쓰기
- `저렴하게 해드릴 수 있습니다`만 쓰고 근거를 적지 않기

### 4. 관련 포트폴리오

선택지는 아래 두 가지를 기준으로 판단합니다.

- 관련 포트폴리오가 있습니다.
- 없습니다.

관련 포트폴리오가 있을 때 작성 기준:

- 현재 지원 프로젝트와 가장 가까운 사례 1~3개만 고릅니다.
- 단순 나열보다 어떤 점이 유사한지 한 줄 설명을 붙입니다.
- 업종, 기능, 전환 동선, 반응형, 운영 경험 중 무엇이 맞닿는지 명확히 적습니다.

예시:

```text
- 제천남여헬스클럽: 반응형 랜딩 페이지, 문의 전환 CTA, SEO 구조 설계 경험
- 새영동숯불갈비: 매장형 브랜드 사이트, 메뉴/위치/예약 동선 설계 경험
- MOUMOUTE: 이커머스 퍼블리싱, 상품 상세 인터랙션, 모바일/데스크탑 UI 개선 경험
```
