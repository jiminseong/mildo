# PRD

## 1. Product Summary

- 프로젝트명: 심리검사 서비스 프로토타입
- 문서 버전: v0.1
- 작성일: 2026-03-14
- 목적: 기존 엑셀 기반 심리검사 도구를 웹 환경에서 실행 가능한 연구용 프로토타입으로 전환하고, 검사 진행, 점수 계산, 결과 리포트, 관리자 로그 확인까지 하나의 제품 흐름으로 검증한다.

이 문서는 `wishket/심리검사서비스프로토타입개발/proposal-demo`를 실제 제품 구조에 가깝게 재구성하기 위한 기준 문서다. 제안용 데모이지만, 구현 방식은 "예쁜 소개 페이지"가 아니라 "실제 프로젝트의 1차 제품 베이스"를 목표로 한다.

## 2. Problem Statement

- 현재 검사는 아날로그 문항지와 엑셀 계산식 중심으로 운영되고 있다.
- 응답자는 온라인에서 바로 검사에 참여할 수 없고, 결과 산출 과정이 수동적이다.
- 연구용 반복 실험이나 파일럿 운영 시 접속 로그, 완료율, 결과 리포트 재현성이 떨어진다.
- 상용화 이전 단계에서 가장 먼저 검증해야 하는 것은 UI보다 "엑셀 로직을 웹 계산 엔진으로 안정적으로 옮길 수 있는가"다.

## 3. Product Goals

- 회원가입 없이 공개 URL로 검사에 즉시 진입할 수 있어야 한다.
- 문항 응답 후 웹 화면에서 즉시 결과 리포트를 생성해야 한다.
- 엑셀 계산 로직을 웹 계산 모듈로 분리 가능한 구조를 제시해야 한다.
- 리포트 저장과 관리자 로그 조회를 포함한 연구 프로토타입 운영 흐름을 보여줘야 한다.
- 제안 단계에서도 향후 실제 개발로 확장 가능한 화면 구조와 코드 구조를 확보해야 한다.

## 4. Non-Goals

- 임상 진단용 고정 PDF 양식 확정
- 실제 통계 엔진 및 엑셀 업로드 연동
- 운영용 관리자 인증 체계
- 개인정보 수집 정책, 약관, 법적 고지 확정
- 상용 서비스 수준의 보안, 권한, CMS

## 5. Primary Users

### 5.1 검사 참여자

- 회원가입 없이 링크로 진입한다.
- 순서대로 문항에 응답한다.
- 검사 완료 후 결과 리포트를 확인하고 저장한다.

### 5.2 연구 담당자 / 운영자

- 세션별 접속 로그와 완료 여부를 확인한다.
- 응답 데이터와 결과 산출 구조를 검토한다.
- 향후 연구용 통계 처리 범위를 판단한다.

### 5.3 클라이언트 / 발주처

- 엑셀 기반 기존 도구가 웹 서비스로 어떻게 전환되는지 본다.
- 실제 구축 시 어떤 화면과 기술 구조가 필요한지 판단한다.
- 프로토타입 이후 본개발 범위를 협의한다.

## 6. User Scenarios

### 시나리오 A. 공개 진입형 검사

1. 사용자가 공개 URL에 접속한다.
2. 검사 개요와 유의사항을 확인한다.
3. 응답자 이름 또는 별칭을 입력하고 검사를 시작한다.
4. 문항에 순차 응답한다.
5. 완료 시 결과 리포트로 이동한다.

### 시나리오 B. 결과 리포트 확인 및 저장

1. 사용자가 완료된 세션의 결과 요약을 본다.
2. 척도별 점수와 해석 문구를 확인한다.
3. 브라우저 인쇄를 통해 PDF 저장을 실행한다.

### 시나리오 C. 운영자 로그 확인

1. 운영자가 관리자 화면에 접속한다.
2. 세션별 시작 시각, 완료 여부, 리포트 생성 상태를 확인한다.
3. 향후 저장 항목과 연구 통계 처리 범위를 검토한다.

## 7. Functional Requirements

### 7.1 공개 진입 홈

- 프로젝트 목적, 데모 범위, 핵심 모듈을 설명한다.
- 검사 시작, 결과 보기, 관리자 로그 확인 진입점을 제공한다.
- 현재 세션 상태를 한눈에 보여준다.

### 7.2 검사 진행 화면

- 문항은 1개씩 표시한다.
- 5점 척도 응답을 제공한다.
- 이전/다음 문항 이동이 가능해야 한다.
- 진행률과 응답 현황을 보여준다.
- 응답 중에도 실시간으로 점수 요약이 계산된다.
- 예시 응답 채우기와 세션 초기화가 가능해야 한다.

### 7.3 결과 리포트 화면

- 세션 메타정보를 보여준다.
- 척도별 T-score, 원점수, 상태 구간, 해석 문구를 표시한다.
- 전체 요약 해석을 생성한다.
- PDF 저장 예시를 위해 브라우저 인쇄 플로우를 제공한다.
- 검사 미완료 상태에서도 초안 리포트 미리보기를 제공한다.

### 7.4 관리자 화면

- 접속 로그 테이블을 제공한다.
- 현재 세션과 샘플 세션을 함께 확인할 수 있어야 한다.
- 연구용 통계 스냅샷과 기술 이관 체크리스트를 표시한다.
- 실개발 시 서버 저장 구조와 이벤트 로그 범위를 논의할 수 있게 해야 한다.

### 7.5 공통 가이드

- 모든 화면에서 가이드 버튼을 제공한다.
- 가이드에는 데모 목적, mock 범위, 시연 방법, 확인 포인트를 포함한다.

## 8. Information Architecture

```text
/                홈 / 제안 개요
/assessment      공개 검사 진행 화면
/report          결과 리포트 화면
/admin           관리자 로그 및 연구 운영 화면
```

## 9. Data and Domain Model

### 9.1 핵심 엔티티

- `Question`
  - 문항 ID
  - 문항 텍스트
  - 척도 키
  - 역채점 여부
- `ScaleConfig`
  - 척도명
  - 평균, 표준편차
  - 점수 구간별 해석 문구
- `Session`
  - 세션 ID
  - 응답자 별칭
  - 그룹명
  - 시작 시각 / 수정 시각
  - 답변 배열
  - 완료 상태
- `AccessLog`
  - 세션 ID
  - 접속 시작 시각
  - 응답 진행률
  - 리포트 상태
  - 운영 메모

### 9.2 계산 규칙

- 문항 응답은 1~5점 척도
- 역채점 문항은 `6 - answer`
- 척도별 원점수는 소속 문항 점수 합
- 표준점수는 `50 + 10 * ((raw - mean) / std)`
- 결과 구간은 `추가 관찰`, `평균 범위`, `안정 범위 상단`으로 구분

## 10. UX Principles

- 연구 도구처럼 차분하고 신뢰감 있는 편집형 인터페이스를 유지한다.
- 검사를 시작하는 데 장애가 없어야 하며, "설명보다 진행"이 우선되어야 한다.
- 리포트는 임상 진단서처럼 과장하지 않고 연구용 해석 초안으로 제시한다.
- 운영 화면은 화려한 대시보드보다 데이터 확인과 구조 논의에 집중한다.

## 11. Technical Direction

### 11.1 Frontend

- Next.js App Router
- React 19
- TypeScript
- Tailwind CSS 4

### 11.2 Demo Architecture

- 공통 앱 셸: 네비게이션, 가이드 버튼, 세션 상태 표시
- 세션 상태: 브라우저 `localStorage` 기반 저장
- 점수 계산: UI와 분리된 순수 함수 모듈
- 리포트 출력: 브라우저 인쇄 기반 PDF 저장 예시
- 관리자 로그: 목업 로그 + 현재 브라우저 세션 결합 표시

### 11.3 Real Build Expansion

- 서버 저장소: PostgreSQL 또는 Supabase
- 계산 모듈: 엑셀 공식 검증용 테스트 케이스 포함
- PDF: 서버 렌더링 또는 HTML-to-PDF 전략 비교 필요
- 통계 처리: 단일 응답자 리포트와 연구용 통계 분석을 분리 설계

## 12. Delivery Scope

### Phase 1. 분석 및 검증

- 문항/척도 정의서 작성
- 엑셀 시트 구조와 계산 공식 분석
- 예외 규칙과 누락 처리 방침 확정

### Phase 2. 핵심 프로토타입

- 공개 검사 화면
- 결과 리포트
- 로그 조회 화면
- 목업 데이터 및 계산 엔진 1차 이관

### Phase 3. 운영형 확장

- 서버 저장
- 관리자 인증
- PDF 양식 고정
- 통계 처리 고도화

## 13. Risks and Open Questions

- 실제 엑셀 파일의 시트 구조와 수식 복잡도가 아직 공개되지 않았다.
- 통계 처리 범위가 단순 요약인지, t-test 포함 연구 분석인지 확정이 필요하다.
- 리포트 PDF가 인쇄형인지, 정형 템플릿인지 결정이 필요하다.
- 개인정보 저장 여부와 익명 운영 정책에 따라 데이터 모델이 크게 달라질 수 있다.

## 14. Success Metrics

- 공개 진입 후 검사 시작까지 30초 이내
- 파일럿 완료율 80% 이상
- 엑셀 결과와 웹 결과 간 샘플 검증 오차 0
- 운영자가 세션 상태와 결과 생성 여부를 즉시 파악 가능
