# 중고등학생 대상 특허/발명 아이디어 고도화 교육 웹 서비스 지원서 초안

## 기본 입력값

- 지원 금액: 9,000,000원
- 지원 기간: 60일
- 관련 포트폴리오: 있습니다

## 제출용 본문

```text
<프로젝트 진행 제안>
- 본 프로젝트는 학생용 학습 플로우와 관리자 승인 플로우가 결합된 반응형 교육 웹 서비스로 이해했습니다.
- 공고 기준 핵심 기능은 아래와 같습니다.
  - 이름 + 일회성 접속코드 기반 로그인
  - 개인정보 수집 및 발명가 번호 발급
  - 진단 결과에 따른 단계 분기
  - 카드 기반 아이디어 고도화
  - 익명 질문·피드백
  - 관리자 승인 후 다음 단계 진행
- 이 프로젝트는 기능 수보다 상태값과 권한 설계가 더 중요해 보입니다.
- 학생의 현재 단계, 저장 데이터, 관리자 승인 시점, 익명 리스트와 실제 사용자 정보 분리 정책을 초기에 정리해야 개발과 운영이 안정적입니다.
- 지원 금액 900만 원은 무리하게 낮춘 금액이 아니라, 현재 공고 범위를 기준으로 1차 구축 범위를 MVP로 명확히 잡았을 때 실제 진행 가능한 금액으로 판단했습니다.
- 1차 구축 범위는 아래까지로 보고 있습니다.
  - 회원/인증
  - 학생 정보 입력
  - 튜토리얼/진단
  - 단계형 아이디어 작성 및 카드 선택
  - 질문·피드백
  - 관리자 조회/승인
  - PC / Mobile / Tablet 반응형
- 고급 통계 대시보드, 복잡한 CMS, 영상 콘텐츠 자체 제작 영역은 2차 고도화로 분리하는 편이 적절합니다.
- 기술 스택은 Next.js + TypeScript + Tailwind CSS 기반 프론트엔드, Supabase(PostgreSQL/Auth/Storage) 기반 백엔드를 우선 제안드립니다.
- 학생/관리자 권한 분리, 발명가 번호 중복 방지, 단계별 데이터 저장, 승인 상태 관리에 적합한 구조로 설계하고 운영 편의를 위한 관리자 화면도 함께 정리하겠습니다.
- 진행 순서는 아래와 같이 제안드립니다.
  - 1주차: 요구사항/정책 정리
  - 2주차: 화면 설계
  - 3~4주차: 디자인 및 핵심 플로우 구현
  - 5~7주차: 학습/피드백/관리자 기능 개발
  - 8주차: QA / 수정 / 배포
- 착수 전에는 아래 항목을 먼저 확인하면 일정과 정책을 더 정확히 확정할 수 있습니다.
  - OTP 발급 방식
  - 카드/교보재 원본 데이터 형식
  - 관리자 승인 시점
  - 개인정보 보관 정책
  - 학부모 연락처 활용 범위
- 진행 중에는 주간 단위로 데모와 문서를 공유하고, 산출물은 화면 설계서, 기능 정의서, 디자인 원본, 개발 소스 기준으로 정리 가능합니다.

<유사 프로젝트 진행 경험>
- 직접적인 교육 플랫폼 납품 사례를 과장해서 말씀드리진 않겠습니다.
- 다만 AIM 앱 패밀리를 직접 기획·설계·개발·운영하면서 아래 경험을 전반적으로 수행했습니다.
  - 사용자 인증
  - 데이터 구조 설계
  - 단계형 사용자 경험 설계
  - 운영 기준 수립
- Supabase 기반 인증과 데이터 구조, 서비스별 권한 분리, 사용자별 기능 흐름 설계를 직접 잡아본 경험이 있어 이번 프로젝트의 학생/관리자 구조 설계에도 바로 적용 가능합니다.
- 제천남여헬스클럽과 새영동숯불갈비 프로젝트에서는 기획·디자인·개발을 일괄 수행하며 반응형 웹, 정보 구조 설계, 실제 운영 가능한 웹사이트 구축을 진행했습니다.
- 이번 프로젝트에서도 PC / Mobile / Tablet 대응과 정보 구조 정리는 같은 기준으로 진행할 수 있습니다.
- 이번 공고에서 특히 중요한 부분은 아래 3가지라고 보고 있습니다.
  - 익명 발명가 번호 기반 공유
  - 14~15단계 수준의 반복형 아이디어 고도화
  - 관리자 승인 후 다음 단계 진행
- 이 부분은 단순 화면 구현보다 데이터 모델과 상태 전이 설계가 핵심이기 때문에, 초기 화면 설계 단계에서 정책과 함께 묶어서 정리하겠습니다.

<사용 기술과 툴>
- 프론트엔드: Next.js, React, TypeScript, Tailwind CSS
- 백엔드/DB: Supabase, PostgreSQL, Storage, 권한 정책(RLS) 또는 필요 시 별도 API 서버 구조 제안
- 작업 방식: 요구사항 정의와 정책 정리를 먼저 문서화한 뒤, 화면 설계와 디자인, 개발을 단계적으로 병행합니다.
- 커뮤니케이션: 위시켓 메시지와 정기 미팅 기준으로 진행하고, 확인이 필요한 이슈는 문서와 데모 화면으로 빠르게 정리해 의사결정 부담을 줄이겠습니다.
- 산출물 정리: 화면 설계서, 기능 명세, 디자인 원본, 소스 코드, 운영/배포 가이드를 정리해 전달 가능합니다.

관련 포트폴리오:
- AIM 앱 패밀리: 사용자 인증, 데이터 구조 설계, 권한 분리, 단계형 사용자 흐름을 직접 설계·운영한 자체 서비스 프로젝트
- 제천남여헬스클럽: 반응형 웹, 정보 구조 설계, 실제 운영 가능한 서비스 화면 구성 경험
- 새영동숯불갈비: 기획·디자인·개발을 한 번에 진행하며 사용자 동선과 반응형 UX를 정리한 웹 구축 사례
```
