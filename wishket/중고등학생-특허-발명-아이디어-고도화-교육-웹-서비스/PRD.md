# 아이디어 워크숍 — 제품 요구사항 정의서 (PRD)

## 1. 제품 정의

**서비스명**: 아이디어 워크숍

**한 줄 설명**: 중고등학생이 특허·발명 아이디어를 단계적으로 발전시키고, 운영자가 승인과 피드백으로 품질을 관리하는 교육 웹 서비스

**배경**: 기존 오프라인 교보재 중심의 발명 교육은 학생 진행 상태 추적이 어렵고, 학생 간 아이디어 교류가 제한적이며, 운영자의 승인·피드백 흐름이 비효율적이다. 이를 웹 기반 학습 도구로 전환하여 학생이 스스로 따라가며 아이디어를 고도화하고, 운영자가 실시간으로 관리할 수 있는 환경을 만든다.

**핵심 가치**:
- 학생은 순서대로 따라가기만 하면 초안 → 중간안 → 최종안까지 아이디어가 정리된다.
- 발명가 번호를 통해 학생 간 익명 피드백 교환이 가능하다.
- 운영자는 승인/보완 요청으로 학생 진행을 관리하며, 승인 결과가 학생 화면에 즉시 반영된다.

**기술 스택**: Next.js 16, React 19, TypeScript, Tailwind CSS 4

**현재 단계**: 프론트엔드 구현 (백엔드는 React 상태로 시뮬레이션, 추후 Supabase 연동 예정)

---

## 2. 설계 원칙

1. **실제 서비스 수준의 완성도**: 사용자가 보는 모든 화면은 실제 교육 현장에서 바로 쓸 수 있는 수준이어야 한다. "시연용", "프로토타입" 같은 표현은 사용하지 않는다.
2. **학생 중심 UX**: 중고등학생이 별도 설명 없이도 다음에 뭘 해야 하는지 바로 알 수 있어야 한다.
3. **단계 잠금 구조**: 이전 단계를 완료해야 다음 단계가 열리는 구조를 통해 학습 품질을 보장한다.
4. **익명성 보장**: 게시판과 피드백에서는 발명가 번호만 노출되며, 실명은 운영자만 볼 수 있다.
5. **반응형**: 데스크톱, 태블릿, 모바일에서 동일한 기능을 제공한다.
6. **프리미엄 디자인**: 교육 서비스이지만 세련된 느낌을 주어야 한다. 군더더기 없는 카드 레이아웃, 부드러운 전환, 정돈된 타이포그래피를 사용한다.

## 3. 사용자 역할

### 3-1. 학생
- 이름과 접속코드로 로그인한다.
- 개인정보를 입력하고 발명가 번호를 발급받는다.
- 15단계 학습 흐름을 따라가며 아이디어를 발전시킨다.
- 익명 게시판에서 다른 학생의 아이디어를 보고 피드백을 교환한다.
- 운영자 승인 후 다음 단계로 진행한다.
- 최종 퀴즈를 풀고 학습을 완료한다.

### 3-2. 운영자
- 학생 목록, 프로필, 현재 단계, 제출 데이터를 확인한다.
- 중간 승인과 최종 승인을 처리한다 (승인 / 보완 요청).
- 승인 결과는 학생 화면의 단계 잠금에 즉시 반영된다.
- 익명 게시판과 피드백 현황을 모니터링한다.

## 5. 학생 워크스페이스 (15단계 학습 흐름)

### 레이아웃
- **좌측 사이드바**: 15단계 목록. 현재 단계 하이라이트, 완료/잠금 상태 아이콘 표시, 발명가 번호·현재 단계·아이디어 상태 요약 카드
- **우측 메인 영역**: 현재 단계의 상세 컨텐츠

### 단계 상세

**1단계 — 로그인**
- 학생 이름 + 접속코드 입력
- 접속코드 확인 후 세션 시작
- 잘못된 코드 시 안내 메시지

**2단계 — 개인정보 입력**
- 성별, 생년, 학년, 학교, 지역, 보호자 연락처, 개인정보 수집 동의
- 제출 시 발명가 번호 자동 생성
- 발명가 번호는 이후 모든 화면에서 익명 식별자로 사용

**3단계 — 학습 안내**
- 서비스 사용법과 핵심 개념 안내
- 발명가 번호의 역할, 피드백 방식, 단계 승인 구조 설명
- 안내 완료 후 다음 단계 활성화

**4단계 — 진단**
- "이미 아이디어가 있다 / 아직 없다" 선택
- 주제 분야, 해결하고 싶은 문제, 진행 트랙 입력
- 아이디어가 없는 경우: 주제 기반 추천 초안 자동 생성

**5단계 — 초안 작성**
- 아이디어 제목, 한 줄 요약, 문제 정의, 대상 사용자, 초기 인사이트 입력
- 실시간 미리보기 카드 제공
- 저장 시 익명 게시판에 초안 등록

**6단계 — 카드 탐색**
- 과학 원리 / 기술 요소 / 발명 원리 / 해결 질문 4종 카드 그룹
- 카드를 선택해 아이디어와 연결 (최소 4장)
- 선택 현황이 우측 요약에 실시간 반영

**7단계 — 익명 게시판**
- 발명가 번호 기반 아이디어 목록
- 초안 / 중간안 / 최종안 필터
- 다른 학생 아이디어 상세 보기 + 익명 피드백 작성
- 피드백 1회 이상 작성해야 다음 단계 활성화

**8단계 — 피드백 반영**
- 내 아이디어에 들어온 피드백 목록
- 각 피드백에 답변 작성
- 수정 메모 작성
- 모든 답변 + 수정 메모 완료 시 다음 단계 활성화

**9단계 — 중간 제출**
- 피드백 반영 결과를 정리한 수정 제목 + 수정 해결안 작성
- 제출 전 체크리스트 (피드백 작성 여부, 답변 완료 여부 등)
- 제출 시 운영자 1차 승인 대기 상태로 전환

**10단계 — 1차 승인 대기**
- 현재 승인 상태 표시 (대기 / 승인 / 보완 요청)
- 운영자 메모 표시
- 승인 시 → 11단계 활성화
- 보완 요청 시 → 9단계로 돌아가 수정

**11단계 — 솔루션 구조화**
- 스토리보드 작성
- 구현 절차 작성
- 아이디어를 실행 가능한 형태로 정리

**12단계 — 자료 첨부**
- 이미지 파일 업로드 (미리보기)
- 자료 설명 메모 작성
- 스케치, 도면, 참고 자료 등 첨부

**13단계 — 최종 제출**
- 최종 피치 문구 작성
- 기대 효과 작성
- 제출 시 운영자 최종 승인 대기 상태로 전환
- 보완 요청 시 수정 후 재제출 가능

**14단계 — 마무리 퀴즈**
- 운영자 최종 승인 후 활성화
- 객관식 3문항 이상
- 제출 시 점수 계산

**15단계 — 완료**
- 전체 학습 결과 요약
- 발명가 번호, 아이디어 상태, 퀴즈 점수, 선택 카드 표시

## 6. 운영자 대시보드

### 레이아웃
2단 레이아웃:
- **좌측**: 승인 대기 큐 + 현재 학생 정보
- **우측**: 학생 목록 + 게시판 현황

### 기능 상세

**대시보드 요약**
- 총 학생 수, 승인 대기 건수, 게시판 아이디어 수

**승인 대기 큐**
- 1차 승인 / 최종 승인 대기 목록
- 학생 발명가 번호, 제출 단계, 제출 내용 요약
- 승인 메모 입력 → 승인 / 보완 요청 처리
- 처리 결과가 학생 화면에 즉시 반영

**학생 정보**
- 실명, 발명가 번호, 학교, 학년, 현재 단계
- 운영자만 실명-발명가 번호 매핑을 볼 수 있음

**학생 목록**
- 등록된 학생 카드 (발명가 번호, 현재 단계, 상태)

**게시판 현황**
- 익명 게시판에 등록된 아이디어 리스트 요약

## 9. 단계 전이 규칙

| 단계 | 전이 조건 |
|------|-----------|
| 1→2 | 이름 + 접속코드 확인 |
| 2→3 | 개인정보 전체 입력 + 동의 |
| 3→4 | 학습 안내 완료 |
| 4→5 | 진단 항목 전체 입력 |
| 5→6 | 초안 5개 항목 작성 |
| 6→7 | 카드 4장 이상 선택 |
| 7→8 | 피드백 1회 이상 작성 |
| 8→9 | 모든 피드백 답변 + 수정 메모 작성 |
| 9→10 | 수정 제목 + 수정 해결안 제출 → 승인 대기 |
| 10→11 | 운영자 1차 승인 |
| 11→12 | 스토리보드 + 구현 절차 작성 |
| 12→13 | 자료 업로드 + 설명 메모 작성 |
| 13→14 | 최종 피치 + 기대효과 제출 → 운영자 최종 승인 |
| 14→15 | 퀴즈 전체 답변 제출 |

**승인 규칙**:
- 10단계: 운영자가 승인하면 11단계 활성화. 보완 요청하면 9단계로 복귀.
- 13단계: 운영자가 최종 승인하면 14단계 활성화. 보완 요청하면 수정 후 재제출.

## 11. 현재 구현 범위와 향후 계획

### 현재 (프론트엔드)
- 모든 UI와 인터랙션은 완전히 동작
- 데이터는 브라우저 메모리에서 관리 (새로고침 시 초기화)
- 접속코드는 고정 목록으로 확인
- 이미지 업로드는 로컬 미리보기만 지원

### 향후 (백엔드 연동 시)
- Supabase 기반 사용자 인증 및 데이터 저장
- 실시간 알림 (승인 완료, 피드백 수신 등)
- 파일 업로드를 클라우드 스토리지로 전환
- 운영자 권한 시스템 및 멀티 세션 지원
- 학습 통계 대시보드

## 12. 수용 기준

1. 학생 화면에서 1단계부터 15단계까지 실제 입력과 클릭으로 이동 가능해야 한다.
2. 운영자 화면에서 승인/보완 요청 시 학생 진행 상태가 즉시 변경되어야 한다.
3. 익명 게시판 필터, 카드 선택, 피드백 작성/답변, 파일 업로드 미리보기가 모두 동작해야 한다.
4. 모든 페이지에서 헤더와 푸터가 일관되게 표시되어야 한다.
5. 모바일, 태블릿, 데스크톱에서 레이아웃이 무너지지 않아야 한다.
6. 어떤 화면에서도 "데모", "프로토타입", "시연", "mock", "예시 데이터" 같은 표현이 노출되지 않아야 한다.
7. `npm run lint`, `npm run build`를 통과해야 한다.
